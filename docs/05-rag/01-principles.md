# 5.1 RAG 原理详解 —— 检索增强生成的核心机制

## 📖 导读

> **LLM 的知识止于它的训练数据。RAG 让 LLM 能"看到"训练数据之外的世界。**

RAG（Retrieval-Augmented Generation，检索增强生成）是当前最主流的 LLM 应用范式。它通过**先检索相关信息，再基于检索结果生成答案**的方式，解决了 LLM 的"知识截止日期"问题，让 AI 可以基于你自己的文档回答问题。

## 二、为什么需要 RAG？

### 2.1 LLM 的固有局限

```text
LLM 就像一个知识渊博但不更新的教授：
✅ 懂得很多（训练数据中的知识）
❌ 不知道新知识（训练后的新信息）
❌ 不知道你的私有数据（公司文档、个人信息）
❌ 会产生幻觉（编造不确定的信息）
```

### 2.2 解决方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RAG** | 检索相关文档 + LLM 基于文档回答 | ✅ 实时更新 ✅ 可溯源 ✅ 低成本 | ❌ 依赖检索质量 |
| **微调** | 用新数据重新训练模型 | ✅ 深度融入模型 | ❌ 成本高 ❌ 更新慢 ❌ 可能遗忘 |
| **长上下文** | 把所有文档塞进 context | ✅ 简单直接 | ❌ 成本极高 ❌ 受窗口大小限制 |

### 2.3 RAG 的核心价值

```text
❌ 没有 RAG：
用户：我们公司的离职流程是什么？
LLM：抱歉，我无法获取贵公司的内部信息...

✅ 有 RAG：
用户：我们公司的离职流程是什么？
     ↓ 检索公司内部文档
     ↓ LLM 基于文档回答
LLM：根据公司《员工手册》第 3 章，离职流程如下：
1. 提交离职申请给直属上级
2. HR 进行离职面谈
3. ...（引用：员工手册.pdf 第 3 页）
```

### 3.2 两大阶段详细说明

**阶段一：索引（Indexing）—— 建好知识库**

```text
原始文档 → 加载 → 分块 → Embedding → 存入向量库
                                                   ↓
                                           以后检索时用
```

**阶段二：查询（Query）—— 回答问题**

```text
用户问题 → Embedding → 向量库检索 → 找到相关文档
                                        ↓
                              构建 Prompt：文档 + 问题
                                        ↓
                                    LLM 生成答案
```

## 四、RAG 的三种架构

### 4.1 Naive RAG（基础版）

最简单的 RAG：一次检索，一次生成。

```python
def naive_rag(query: str) -> str:
    # 1. 检索
    docs = retriever.invoke(query)
    
    # 2. 增强
    context = "\n\n".join([d.page_content for d in docs])
    prompt = f"基于以下信息回答问题：\n\n{context}\n\n问题：{query}"
    
    # 3. 生成
    return llm.invoke(prompt)
```

**特点**：简单、快速，适合文档质量高的场景。

### 4.2 Advanced RAG（进阶版）

在检索前后增加优化步骤。

```python
def advanced_rag(query: str) -> str:
    # 预处理：查询优化
    rewritten_query = query_rewrite(query)  # 重写模糊查询
    sub_queries = decompose_query(query)   # 分解复杂查询
    
    # 检索
    docs = retriever.invoke(rewritten_query)
    
    # 后处理：重排序
    reranked_docs = reranker.rerank(query, docs)
    
    # 生成
    return generate_with_context(reranked_docs, query)
```

**特点**：质量更高，适合复杂查询和噪声较多的文档。

### 4.3 Modular RAG（模块化版）

将 RAG 拆分为可替换的独立模块。

```python
class ModularRAG:
    """模块化 RAG：每个组件可独立替换"""
    
    def __init__(self):
        self.query_processor = QueryProcessor()    # 查询处理
        self.retriever = HybridRetriever()         # 混合检索
        self.reranker = CohereReranker()           # 重排序
        self.generator = LLMGenerator()            # 生成
        self.validator = AnswerValidator()          # 验证
    
    def answer(self, query: str) -> dict:
        processed = self.query_processor.process(query)
        docs = self.retriever.retrieve(processed)
        ranked = self.reranker.rerank(query, docs)
        answer = self.generator.generate(query, ranked)
        validated = self.validator.validate(answer)
        return validated
```

## 五、RAG vs 微调 vs 纯 LLM

| 维度 | 纯 LLM | RAG | 微调 |
|------|--------|-----|------|
| **知识时效** | 训练截止日期 | ✅ 实时更新 | 训练截止日期 |
| **私有数据** | ❌ 看不到 | ✅ 可以访问 | ✅ 可以融入 |
| **可溯源** | ❌ 黑盒 | ✅ 可追溯来源 | ❌ 黑盒 |
| **幻觉控制** | ❌ 易产生 | ✅ 有据可依 | ⚠️ 略有改善 |
| **实施成本** | 低（直接调用） | 中（建向量库） | 高（需要算力） |
| **更新速度** | 快 | ✅ 秒级更新 | 慢（重新训练） |
| **维护复杂度** | 低 | 中 | 高 |

### 什么时候选 RAG？

- ✅ 你需要回答关于**私有文档**的问题
- ✅ 你需要**实时或近实时**的信息
- ✅ 你需要**可追溯**的回答（知道答案来源）
- ✅ 你没有资源或意愿去微调模型

## 六、RAG 的局限性

| 问题 | 表现 | 解决方案 |
|------|------|----------|
| **检索失败** | 没找到相关文档 | 查询扩展、HyDE 技术 |
| **上下文碎片** | 检索的片段不完整 | 适当增大 chunk、增加 overlap |
| **信息过载** | 太多不相关内容干扰 LLM | 重排序、压缩上下文 |
| **遗漏关键信息** | 重要的文档没被检索到 | 混合检索（词法+语义） |

## 八、本章总结

| 知识点 | 一句话说明 |
|--------|------------|
| **RAG** | 先检索相关文档，再基于文档生成答案 |
| **索引阶段** | 文档→分块→向量化→存储 |
| **查询阶段** | 问题→向量化→检索→增强→生成 |
| **Naive RAG** | 单次检索+单次生成，简单快速 |
| **Advanced RAG** | 增加查询优化和结果重排序 |
| **Modular RAG** | 各模块可独立替换和优化 |
