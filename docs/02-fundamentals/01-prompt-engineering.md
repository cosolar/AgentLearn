# 2.1 Prompt 工程基础 —— 掌握与 LLM 对话的艺术

## 📖 导读

> **相同的模型，不同的 Prompt → 完全不同的结果。**

Prompt Engineering（提示工程）是 Agent 开发者的基础技能，也是决定 Agent 表现上限的关键因素。**一个好的 Prompt 能让 GPT-3.5 击败 GPT-4，一个差的 Prompt 能让 GPT-4 变得不如 GPT-3.5。** 本文将从零开始，系统讲解 Prompt 工程的核心技法、进阶模式，以及大量可复用的实战模板。

---

## 一、前置知识

| 概念 | 说明 |
|------|------|
| **System Prompt** | 系统级指令，设定 AI 的角色、行为规则和输出约束 |
| **User Prompt** | 用户的输入，包含具体问题和指令 |
| **Few-shot** | 在 Prompt 中提供示例，引导模型按示例的模式输出 |
| **Token** | LLM 的最小处理单元，1 个英文字 ≈ 1 token，1 个汉字 ≈ 2-3 tokens |
| **Context Window** | LLM 能处理的输入最大长度（如 GPT-4o 是 128K tokens） |

---

## 二、为什么 Prompt 如此重要？

### 2.1 直观对比

```text
❌ 差的 Prompt:
"写一个故事"

→ 输出可能很简短、普通，没有任何亮点


✅ 好的 Prompt:
"请写一个 800 字的科幻短篇故事，核心创意是'时间可以被购买但不能被创造'，
主人公是一个时间商人。采用第三人称限知视角，开头要有悬念，结尾要有反转。"

→ 输出更有方向、更具体、质量稳定
```

### 2.2 Prompt 的质量差距

| 维度 | 差 Prompt | 好 Prompt |
|------|-----------|-----------|
| **明确性** | 模糊、多义 | 清晰、具体、无歧义 |
| **约束性** | 无约束 | 有格式/长度/风格约束 |
| **上下文** | 缺少背景 | 提供完整上下文 |
| **示例** | 无示例 | 有准确的 few-shot 示例 |
| **结构** | 一段文字 | 分部分、分层级 |

## 三、基础 Prompt 技巧

### 3.1 Zero-shot Prompting（零样本提示）

不给任何示例，直接让模型完成任务。适用于**简单、常见的任务**。

```text
# 示例 1：情感分类
请判断以下文本的情感是"正面"还是"负面"：
"这部电影太精彩了，我看了三遍！"
输出：正面

# 示例 2：文本总结
用一句话总结以下内容：
[文章内容]
```

**适用场景**：情感分析、简单分类、基础问答

**局限性**：对于复杂或罕见任务，Zero-shot 的效果不稳定。

### 3.2 Few-shot Prompting（少样本提示）

提供 2-5 个示例，让模型"照猫画虎"。适用于**格式要求严格或模式复杂的任务**。

```text
# 示例：意图分类
输入：今天天气真好
输出：闲聊

输入：帮我查一下订单号 12345
输出：查询

输入：我要投诉你们的产品
输出：投诉

输入：价格实惠，质量不错
输出：？
```

**Few-shot 的最佳实践**：

| 示例数量 | 效果 | 说明 |
|----------|------|------|
| 0 (zero-shot) | 基线 | 对于简单任务已足够 |
| 1-3 | 良好 | 大多数任务的最佳区间 |
| 3-5 | 优秀 | 复杂任务需要更多示例 |
| 5+ | 边际递减 | 过多的示例反而会分散模型注意力 |

### 3.3 Chain-of-Thought (CoT)（思维链提示）

引导模型展示推理过程，**先讲逻辑再给答案**。适用于**数学、逻辑、推理类任务**。

```text
# ❌ 普通 Prompt
小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有几个？

# ✅ CoT Prompt
小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有几个？
请逐步推理。

思考过程：
1. 初始有 5 个苹果
2. 吃了 2 个：5 - 2 = 3 个
3. 又买了 3 个：3 + 3 = 6 个
4. 所以现在有 6 个苹果
答案：6
```

**CoT 的威力**：研究表明，CoT 能让 LLM 在数学推理任务上的准确率提升 20-30%。

### 3.4 角色设定（Role Prompting）

给 LLM 分配一个**专业角色**，激活其在特定领域的知识。

```text
# ✅ 有角色设定
你是一名资深 Python 架构师，专注于高性能 Web 服务，
了解 FastAPI 所有最佳实践和常见陷阱。

请 review 以下代码……

# ❌ 无角色设定
请 review 以下代码……
```

**效果对比**：

| 场景 | 有角色 | 无角色 |
|------|--------|--------|
| 代码审查 | 能指出性能问题、安全漏洞 | 仅检查语法错误 |
| 法律咨询 | 使用专业术语，指出法律条文 | 给出笼统建议 |
| 医疗建议 | 严谨，列明局限性 | 可能给出危险建议 |

## 四、进阶 Prompt 模式

### 4.1 ReAct 模式（Reasoning + Acting）

让模型**边思考边行动**，是 Agent 的基础模式。详见 2.3 节。

```text
你是一个可以调用工具的 AI 助手。
请按照以下格式思考并行动：

问题：用户的问题是什么？
思考：我需要做什么才能回答这个问题？
行动：调用某个工具（如 search_web、calculate）
观察：工具返回的结果是什么
思考：根据观察结果，我是否已经能回答？
回答：最终的答案
```

### 4.2 Plan-and-Solve（先规划后执行）

先制定完整计划，再逐步执行。适用于**复杂多步骤任务**。

```text
请按以下步骤完成这篇技术文章的写作：

第一步：制定大纲
- 列出文章的核心论点
- 确定每个章节的子标题

第二步：逐章写作
- 按大纲顺序，每章写 300-500 字

第三步：全局修改
- 检查逻辑连贯性
- 统一术语和风格

现在，开始第一步：制定大纲。
```

### 4.3 Self-Refine（自我优化）

让模型先输出，然后**自己检查和优化**自己的输出。

```text
第一步：生成初稿
请写一段关于 AI Agent 的介绍，200 字左右。

第二步：自我审查
请检查上述内容是否存在以下问题：
1. 是否有技术不准确的地方？
2. 是否有逻辑不连贯的地方？
3. 是否有可读性差的地方？

第三步：优化
请根据你的审查结果，生成优化后的版本。
```

### 4.4 Structured Output（结构化输出）

指定输出格式，让 LLM 的回复**可以被程序直接解析**。

```python
# 使用 JSON 格式输出
prompt = """
分析以下产品的特点，以 JSON 格式输出：

产品：iPhone 15 Pro

输出格式：
{
    "name": "产品名",
    "category": "类别",
    "key_features": ["特点1", "特点2", "特点3"],
    "pros": ["优点1", "优点2"],
    "cons": ["缺点1", "缺点2"],
    "rating": 评分(1-10)
}
"""
```

### 4.5 Persona Cascade（多重人格级联）

在一个对话中让 LLM 扮演**多个角色**依次处理。

```text
你现在是一个三位一体的专家团队：

1. 🧑‍💻 工程师：评估技术可行性
2. 🎨 设计师：评估用户体验
3. 📊 产品经理：评估商业价值

我将给你一个产品想法，请三位依次给出意见。
最后，产品经理综合三位意见给出最终建议。

产品想法：一个用 AI 自动生成代码注释的工具
```

## 五、Prompt 模板库（可直接复用）

### 5.1 代码审查 Prompt

```text
你是一名资深代码审查员，请 review 以下代码：

任务：
1. 找出潜在 bug 和逻辑错误
2. 指出性能问题
3. 检查代码风格和最佳实践
4. 指出安全隐患

输出格式：
- 严重问题（必须修复）：
- 建议改进（推荐修改）：
- 风格建议（可选）：
- 总结（总体评价 + 修改优先级）：

代码：
```python
{你的代码}
```
```

### 5.2 知识问答 Prompt

```text
你是一名专业的技术导师。请回答以下问题：

要求：
1. 先用一句话给出核心答案
2. 然后分点详细解释
3. 如果可能，提供代码示例
4. 指出常见的误解或踩坑点

问题：{用户问题}
```

### 5.3 文本总结 Prompt

```text
请对以下内容进行总结：

【任务】
1. 用 3 句话概括核心内容
2. 列出 3-5 个关键要点（bullet points）
3. 如果文中包含数据或数据，在总结中保留

【原文】
{文本内容}

【输出格式】
核心概括：
{一句话}

详细总结：
{3句话}

关键要点：
- 要点1
- 要点2
...

原文数据：
{保留关键数据}
```

## 六、Prompt 优化 checklist

写 Prompt 时，对照这个清单逐一检查：

| 检查项 | 说明 |
|--------|------|
| ✅ **目标明确** | LLM 知道你要什么吗？ |
| ✅ **约束充分** | 格式、长度、风格定了吗？ |
| ✅ **上下文完整** | 背景信息给够了吗？ |
| ✅ **边界清晰** | 什么能做、什么不能做说清了吗？ |
| ✅ **示例准确** | 如果需要示例，示例够典型吗？ |
| ✅ **输出可解析** | 如果需要程序处理，格式是指定好的吗？ |

### 常见错误

```text
❌ 过于模糊
"总结这篇文章"

✅ 清晰具体
"用 3 句话总结这篇文章，重点突出作者的核心论点，
输出使用中文，每句话不超过 50 字"

❌ 信息过载
一次把 10 个要求塞进一个 Prompt

✅ 结构清晰
分步骤、分模块地提供信息

❌ 缺少约束
"写一篇关于 AI 的文章"

✅ 有约束
"写一篇 2000 字左右的技术博客，面向有 1-3 年经验的开发者，
主题是 RAG 技术实践，要求包含代码示例和性能对比数据"
```

| Prompt 模式 | 最佳场景 | 复杂度 |

## 📝 课后练习

1. **📝 比较题**：针对同一任务（如"用 Python 写一个计算器"），分别用 Zero-shot 和 Few-shot（提供 3 个代码示例）写 Prompt，对比输出质量
2. **💡 改造题**：把你之前写的某个 Prompt 按照本文的 Checklist 优化一遍，记录优化前后的输出差异
3. **🔍 探索题**：尝试使用 CoT 解决一个逻辑推理题（如"三个人中谁是说谎者"），观察 CoT 对准确率的影响
4. **🚀 挑战题**：设计一个 Persona Cascade（多重人格级联）Prompt，让 LLM 同时扮演"需求分析师 + 技术选型专家 + 项目风险评估师"来分析你日常工作中的一任务
